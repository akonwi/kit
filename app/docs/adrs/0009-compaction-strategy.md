# 0009: Compaction rewrites persisted session history

## Status
Accepted

## Context

Kit treats persisted `Turn[]` data as the source of truth for session restore, transcript rendering, and future session-level operations.

A compaction feature needs to reduce context pressure in long-running sessions.

There are two broad approaches:

1. keep the full persisted history and apply compaction only as a runtime-only context transform
2. rewrite the persisted session so older turns are replaced by a generated summary turn

The runtime-only approach preserves more raw history, but it introduces a hidden layer between persisted session data and effective model context.

## Decision

Compaction uses a destructive persisted rewrite.

Compaction will:

- summarize older turns
- replace that older prefix with one synthetic summary turn
- preserve a recent suffix of turns verbatim
- save the rewritten `Turn[]` back to the session

After compaction, the rewritten session data becomes the canonical session state.

## Intended shape

The initial intended policy is:

- keep the most recent turns unchanged
- compact older turns into one generated summary turn

The exact retention threshold is still to be decided.

The summary turn should be clearly identifiable as generated by compaction.

## Why this approach

### Positive

- persisted sessions remain the single source of truth
- restore stays simple: load `turns`, rebuild the agent, continue
- transcript rendering stays aligned with what is actually persisted
- no hidden runtime-only context layer needs to be reconstructed later
- compaction becomes an explicit, inspectable session operation

### Trade-off

- compaction mutates session history by replacing detailed older turns with a generated summary

This is acceptable only if compaction is explicit and conservative.

## UX implications

Compaction should remain understandable as a session operation, even if it also runs automatically when context pressure becomes too high.

Expected direction:

- manual compaction should remain explicit and likely use confirmation
- automatic compaction may run without prompting once the session clearly needs it
- result feedback should describe how many turns were compacted and how many were preserved

## Automatic compaction trigger

Automatic compaction runs after a turn completes, never mid-turn.

Policy:

- compute current context usage after the turn is committed
- if usage is known and `percent >= 90`, trigger compaction
- compact before the next normal turn proceeds

## Prerequisite

Before implementing compaction, Kit must first implement context window computation and visibility.

That visibility is needed to answer:

- how full is the current model context?
- when is compaction actually warranted?
- how should the user discover that a session is approaching context limits?

## Consequences

- `/compact` remains deferred for now
- compaction operates on persisted turns, not a hidden runtime-only transform layer
- context-window visibility is a prerequisite feature, not optional polish
