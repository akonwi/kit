# Decision: Compaction rewrites persisted session history

- Status: Accepted
- Date: 2026-04-07

## Context

`kit` now treats persisted `Turn[]` as the source of truth for both:

- session restore
- transcript rendering
- future session-level operations

A possible compaction feature needs to reduce context pressure in long-running
sessions.

There are two broad ways to do that:

1. keep the full persisted history and apply compaction only as a runtime-only
   context transform
2. rewrite the persisted session so older turns are replaced by a generated
   summary turn

The runtime-only approach would preserve more raw history, but it adds an extra
hidden layer to session restore semantics:

- the persisted session would say one thing
- the effective model context would say another
- restore would need to know which compacted state to reconstruct and from what
  point

That conflicts with the current direction of keeping session persistence and
runtime restore behavior explicit and easy to reason about.

## Decision

When `kit` adds compaction, it should use a **destructive persisted rewrite**.

Compaction will:

- summarize older turns
- replace that older prefix with one synthetic summary turn
- preserve a recent suffix of turns verbatim
- save the rewritten `Turn[]` back to the session

After compaction, the compacted session data becomes the canonical session
state. Future restores should load that rewritten turn list directly, with no
separate hidden compacted-context layer.

## Intended shape

The initial intended policy is:

- keep the most recent turns unchanged
- compact older turns into one generated summary turn

The exact retention threshold is still to be decided, but the expected shape is
something like:

- older turns → 1 summary turn
- recent turns → kept exactly as they were

This summary turn should be clearly identifiable as generated by compaction.
That may start as a normal assistant message with explicit summary wording, and
may later grow dedicated metadata if the transcript/runtime need stronger
semantics.

## Why this approach

### Benefits

- persisted sessions remain the single source of truth
- restore stays simple: load `turns`, rebuild `KitAgent`, continue
- transcript rendering stays aligned with what is actually persisted
- no hidden runtime-only context state needs to be reconstructed later
- compaction becomes an explicit, inspectable session operation

### Trade-off

- compaction mutates session history by replacing detailed older turns with a
  generated summary

This is acceptable only if compaction is explicit and conservative.

## UX implications

Compaction should be treated as a deliberate session operation, not an invisible
background optimization.

Expected UX direction:

- user-triggered action
- clear explanation that older history will be replaced by a generated summary
- likely confirmation before proceeding
- result feedback describing how many turns were compacted and how many were
  preserved

## Prerequisite work

Before implementing compaction, `kit` should first implement **context window
computation and visibility**.

We need to be able to answer questions like:

- how full is the current model context?
- when is compaction actually warranted?
- how should the user discover that a session is approaching context limits?

That means the near-term priority is:

1. compute context usage for the active session/model
2. expose that usage in the runtime/UI
3. use that visibility to inform when compaction should be offered or suggested

Compaction should not be implemented before that visibility exists.

## Consequences

- `/compact` should remain deferred for now
- if compaction is implemented later, it should operate on persisted turns, not
  a hidden runtime-only transform layer
- context-window visibility is a prerequisite feature, not optional polish
